## 0. 这版 PRD 解决什么问题

InStock 已经具备不少能力资产：

- A 股数据采集与更新
- 技术指标计算
- K 线形态识别
- 条件筛选
- 策略 / 回测相关实现
- FastAPI + Vue 的全栈工程骨架

问题不在于“缺功能”，而在于：

- 产品定义过宽，容易把任何合理功能都塞进来
- 首页 / 工作台 / AI / 研究沉淀等概念都容易失控
- 路线图如果围绕“平台感”推进，很容易在真正的核心能力做深前就铺摊子

因此，这版 PRD 的首要任务是收敛：

1. 明确 InStock 的长期愿景与当前执行核心不是同一件事
2. 明确什么是核心主线，什么是辅助层
3. 明确哪些内容现在不做，甚至长期都不做
4. 明确产品和架构的每一次迭代都必须回到第一性原理检查

## 1. 第一性原理与长期目标

### 1.1 第一性原理

对个人投资者而言，一个真正有用的 A 股研究产品，不是“信息更多”，而是更高效地支持下面这个循环：

1. **感知市场**
   - 今天市场大概发生了什么？
   - 我的关注对象有没有异常变化？

2. **发现机会**
   - 哪些股票满足我定义的条件？
   - 为什么它们值得进一步看？

3. **验证假设**
   - 这个筛选思路 / 策略在历史上是否有效？
   - 它的收益、风险和适用环境是什么？

4. **持续跟踪**
   - 哪些结果值得继续观察？
   - 哪些条件应该保留、淘汰或调整？

因此，InStock 的核心不是“展示更多”，而是：

> **降低用户从“看到信息”到“形成判断”的摩擦。**

### 1.2 长期愿景

**InStock 的长期愿景是：成为一个面向个人投资者的 A 股研究分析平台。**

它最终应支持：

- 日常市场认知入口
- 个股与板块观察
- 指标 / 形态 / 条件组合筛选
- 策略构建与回测验证
- 研究结果沉淀与跟踪
- AI 作为研究助手的协作

### 1.3 当前执行核心

虽然长期愿景是“研究分析平台”，但当前执行核心必须更收敛：

> **InStock 在当前和未来较长一段时间内，必须优先被当作“面向个人投资者的 A 股量化筛选与策略验证工具”来做。**

原因：

- 这是当前最真实的能力资产所在
- 这是最容易形成差异化的部分
- 这也是最能被用户直接感知价值的主链路

### 1.4 产品判断准则

未来任何需求、页面、架构修改，都必须先回答：

1. 它是否增强了“扫描 -> 筛选 -> 验证”这条核心链路？
2. 它是否缩短了用户从“信息”到“洞察”的路径？
3. 它是否增强了可解释性和可信度？
4. 它是否能沉淀成可复用能力，而不是一次性页面？
5. 它是否会把产品拉向资讯站、交易终端或 AI 荐股器？

如果答案偏负面，应降级或删除。

## 2. 产品定义

### 2.1 最终定义

**InStock 是一个面向个人投资者的 A 股研究分析平台，以量化筛选与策略验证为核心。**

### 2.2 面向用户的核心价值

InStock 的价值不是替用户做决策，而是：

- 更快发现值得研究的股票
- 更清楚理解筛选结果为何成立
- 将个人研究方法沉淀为可复用条件与策略
- 用历史数据验证假设，而不是凭感觉操作
- 在适当时机，用 AI 加速检索、解释和编排，而不是替代底层分析逻辑

### 2.3 不是这些产品

InStock 当前阶段明确不是：

- 券商交易终端
- 资讯门户 / 行情门户
- 股票社区 / 观点社区
- 自动交易平台
- 黑盒 AI 荐股工具

---

## 3. 核心用户与关键任务

### 3.1 核心用户

#### 用户类型 A：主动研究型个人投资者

特征：

- 关注 A 股
- 希望通过条件筛选和技术分析提高研究效率
- 不满足于只看资讯和排行榜
- 希望逐步沉淀自己的研究方法

#### 用户类型 B：策略探索型用户 / 量化爱好者

特征：

- 对条件组合、参数调整、回测验证更敏感
- 更关注策略是否可复现、是否有历史依据
- 愿意先用模板，再逐步走向更复杂表达

### 3.2 用户要完成的关键任务（JTBD）

- 当我打开平台时，我希望快速知道今天市场大致如何，以及我关注的标的是否有异常变化。
- 当我有一个研究想法时，我希望把它变成可运行的筛选条件，而不只是停留在脑子里。
- 当系统筛出一批股票时，我希望知道它们为什么被筛出来，并在图表中看到证据。
- 当我把筛选方法沉淀为策略时，我希望能用历史数据验证它，而不是主观臆断。
- 当我后续还想继续看时，我希望系统能记住我的筛选器、策略、自选和最近研究上下文。
- 当我不想手动翻页面时，我希望 AI 帮我检索、解释和组织下一步动作，而不是替我拍脑袋选股。

## 4. 产品核心主线

InStock 的核心链路必须始终保持为：

**市场感知 -> 条件扫描 -> 结果解释 -> 策略验证 -> 跟踪观察**

这条链路里，真正必须做深的是中间三段：

- 条件扫描
- 结果解释
- 策略验证

“市场感知”只做轻量入口；
“跟踪观察”只做最必要的沉淀，不扩张成泛笔记系统或生产力平台。

## 5. 产品范围与边界

### 5.1 必须做（核心范围）

#### A. 数据与更新能力
- A 股核心数据稳定更新
- 日线/必要频率数据完整性与异常检查
- 数据更新时间可见

#### B. 扫描与筛选能力
- 指标筛选
- 形态筛选
- 参数化条件
- 条件组合（AND / OR / NOT）
- 范围限制（全市场 / 指定板块 / 自选组等）
- 结构化结果输出

#### C. 结果解释能力
- 每个命中结果必须解释“为什么命中”
- 支持在图表中展示命中证据
- 支持快速进入个股详情页

#### D. 个股研究能力
- K 线图与成交量
- 常见技术指标叠加
- 形态识别结果
- 数据时间上下文

#### E. 策略验证能力
- 参数化策略模板
- 历史回测
- 报告页（收益、回撤、胜率、基准对比等）
- 条件 / 策略保存与复用

### 5.2 可以做，但必须克制（辅助范围）

#### 轻量工作台 / 首页
可以做，但只能作为核心引擎的投影面，不能扩张为行情门户。

#### 自选 / 跟踪 / 提醒
可以做，但只服务于核心研究链路。

#### AI 助手
必须晚于核心引擎成熟之后接入，且只做研究辅助。

### 5.3 明确不做（当前阶段，部分内容长期不做）

- 实时 tick 级行情推送
- 交易执行 / 接券商 API
- 新闻 / 公告 / 研报聚合
- 社区 / 评论 / 排行榜
- AI 荐股 / 黑盒推荐
- 多市场（港股/美股）扩张
- SaaS / 多租户 / 复杂权限系统
- Phase 0-1 的低代码策略编辑器
- 以移动端为主的产品方向

## 6. 首页 / 工作台的最小可行设计

### 6.1 首页原则

首页不是一个独立产品模块，而是：

> **核心能力的投影面与导流入口。**

首页上的每一个元素，都必须在 1 次点击内把用户带回：

- 筛选结果
- 个股详情
- 回测 / 策略验证

### 6.2 最小首页模块（只保留 4 个卡片）

#### 卡片 1：市场温度计
展示：
- 上涨 / 下跌家数
- 涨停 / 跌停家数
- 主要指数涨跌
- 一句市场情绪摘要

约束：
- 只用盘后或稳定频率数据
- 不扩张为市场全景页

#### 卡片 2：我的关注
展示：
- 自选股或观察列表
- 更重要的是：其中哪些股票触发了已保存的扫描条件

#### 卡片 3：今日扫描发现
展示：
- 你保存的筛选条件今天命中了多少只股票
- 一键跳到对应结果页

这是首页最关键的卡片，因为它直接服务核心价值。

#### 卡片 4：策略信号 / 回测更新
展示：
- 某个策略新增信号数量
- 最近一次回测更新摘要

此卡可在 Phase 1 后接入。

### 6.3 首页坚决不做

- 板块热力图
- 龙虎榜 / 大单监控
- 新闻 / 公告 / 研报列表
- 多 tab 仪表盘
- 自定义拖拽布局
- 复杂资金流大盘

## 7. 功能模块定义

### 7.1 扫描与筛选引擎（第一核心）

#### 必须支持
- 常见技术指标
- 常见形态识别
- 参数化阈值和周期
- 条件逻辑组合
- 范围过滤
- 结果分页、排序、导出（可后置）

#### 必须输出
- 命中股票列表
- 每只股票的命中原因摘要
- 图表证据可视化入口
- 可保存的筛选条件快照

### 7.2 个股详情（第二核心）

#### 必须支持
- 基础行情摘要
- K 线 + 成交量
- 核心技术指标叠加
- 形态标注
- 当前数据更新时间
- 来自哪些筛选条件的命中说明

### 7.3 策略与回测（第三核心）

#### 当前阶段只做
- 参数化策略模板
- 历史回测
- 回测报告
- 策略参数保存与对比

#### 当前阶段不做
- 通用低代码策略编排器
- 可视化流程编辑器
- 自动交易联动

### 7.4 跟踪与沉淀（辅助核心）

#### 当前阶段只做
- 自选 / 观察分组
- 已保存筛选器
- 已保存策略
- 最近研究上下文
- 必要提醒（可 Phase 2 再强化）

#### 当前阶段不做
- 通用笔记系统
- 复杂项目式研究管理
- 社区化分享沉淀

### 7.5 AI 助手（未来能力）

#### 正确定位
- 检索器
- 解释器
- 编排器
- 摘要器

#### 错误定位
- 选股器
- 拍板器
- 替代规则引擎的黑盒分析器

## 8. 架构要求与技术约束

### 8.1 总体原则

核心引擎必须独立于 Web 框架存在。

也就是说，真正的资产应是：
- data pipeline
- indicator engine
- pattern engine
- screener engine
- strategy / backtest / report engine

而不是 FastAPI router 本身。

### 8.2 存储与执行策略

#### 当前主栈
- 保留 PostgreSQL + Redis 主栈继续迭代
- 不在当前阶段做大规模存储迁移

#### 原因
- 现有工程已沿此方向演进
- 当前阶段的瓶颈不是存储路线，而是产品可交付性
- A 股当前量级下，PG 足够支撑 Phase 0-1

### 8.3 必须预留的两项能力

#### 预留 1：数据访问层抽象
扫描引擎与回测引擎不要直接耦合 ORM/SQL 查询。
应通过 `MarketDataProvider` 等抽象接口获取数据。

#### 预留 2：每日 Parquet 导出
在盘后数据更新链路末端增加 Parquet 导出。
这样未来可平滑引入 DuckDB / Polars / Pandas 作为分析层，而不必重写主引擎。

### 8.4 中期优化方向

如果后续全市场扫描与回测性能成为瓶颈，可在分析层引入：
- DuckDB
- Parquet

但这应是**增量引入**，不是替换主栈。

### 8.5 前端约束

前端最重要的不是首页，而是：
- 筛选面板交互
- 结果列表浏览效率
- 个股详情的证据可视化
- 策略回测报告的可读性

前端状态应尽可能做到：
- URL 即状态
- 刷新不丢上下文
- 可保存/可复现

## 9. AI 的边界与接入前提

### 9.1 为什么不能太早做

#### 原因 1：数据与结果未结构化，AI 只会输出流畅废话
金融语境下，这不是体验问题，而是误导问题。

#### 原因 2：尚未积累足够用户使用模式
在不知道用户最常重复什么动作、最难理解什么结果之前，AI 容易做偏。

#### 原因 3：会导致产品定位漂移
产品很容易从：
- “我设条件，工具帮我找”
滑向：
- “AI 告诉我买什么”

这是必须避免的方向。

### 9.2 AI 接入前提

至少满足：
- 扫描引擎输入输出 schema 化
- 回测报告结构化
- 数据时间上下文可见
- Phase 0-1 完成且稳定运行一段时间

### 9.3 AI 的正确切入点

- 自然语言转筛选条件
- 回测结果解读
- 个股技术面摘要生成
- 下一步操作建议

## 10. 成功标准

### 10.1 Phase 0 成功标准

用户（首先是开发者本人）能够：
- 每天盘后打开 InStock
- 快速运行筛选条件
- 看到命中结果
- 理解命中证据
- 觉得它比直接去资讯终端多出真实价值

### 10.2 Phase 1 成功标准

用户能够：
- 把筛选方法转成参数化策略
- 跑历史回测
- 明确理解收益、风险和基准对比
- 比较不同策略或参数版本的表现

### 10.3 长期成功标准

InStock 真正变成：
- 每日研究入口
- 个人研究方法的承载工具
- 可解释、可信任的筛选与验证平台

而不是另一个“信息很多、但核心价值不够锋利”的股票站点。

## 11. 最终一句话

> **先把“扫描 -> 筛选 -> 验证”这条管道做到自己每天都想用，其他一切都是噪音。**
