# InStock ROADMAP — 产品路线图

> 版本：v2.0 | 更新日期：2026-04-01
> 基于代码实际进度重新评估，将 Phase 0 拆分为 0a/0b
> v1.0 归档：[ROADMAP_v1_archive.md](./ROADMAP_v1_archive.md)

---

## 设计原则

1. **不为理想架构提前支付成本** — 只在需要时引入抽象
2. **每个子阶段必须产出可验证的用户价值** — 不做"纯重构"阶段
3. **对个人项目来说，规划超过 2 个阶段的细节会制造虚假的确定性** — Phase 1 只定义方向，不定义任务

## Phase 0a：最小可用闭环（目标：1-2 周）

### 🎯 里程碑验收标准

> 用户可以在筛选页选择"MACD 金叉 + 成交量放大"条件组合，点击扫描，看到命中结果列表（含命中原因摘要），点击某只股票进入详情页查看 K 线和命中证据。

### 核心任务

**后端（整合为主，非新建）：**
1. 收敛 SelectionService — 合并 SelectionService 和 SelectionServiceWithProvider 为统一服务
2. 标准化命中证据输出 — 扫描结果必须符合 PRD 定义的命中证据数据结构（含 evidence + summary）
3. 个股详情聚合接口 — 整合 stock_service + indicator_service + pattern_service 的数据

**架构（最小必要）：**
4. 确认 engine / service / router 三层边界 — 只确保调用方向单一（router → service → engine/provider）

**前端：**
5. 筛选面板页 — 条件选择 + 参数配置 + 执行扫描
6. 结果列表页 — 命中股票卡片列表，每张卡片显示：股票名称、代码、命中条件摘要、匹配度
7. 个股详情页 v1 — K 线图 + 指标叠加 + 命中证据高亮区域

### 明确不做

- ❌ 首页 / 市场概览（推迟到 0b）
- ❌ 回测功能（推迟到 Phase 1）
- ❌ 数据访问抽象层重构（MarketDataProvider 已存在，保持现状）
- ❌ Parquet 导出（推迟到 Phase 1 回测需要时）
- ❌ 筛选条件保存/加载（推迟到 0b）

### 非功能性门槛（进入 0b 的前提）

| 指标 | 基线要求 |
|------|---------|
| 全市场扫描响应时间 | < 30s（5000 只股票，3 个条件组合） |
| 数据更新任务稳定性 | 连续 5 个交易日成功率 100% |
| 扫描结果正确性 | 随机抽查 10 只命中股票，命中证据与实际数据一致 |

## Phase 0b：日常可用（目标：1-2 周）

### 🎯 里程碑验收标准

> 用户每天盘后打开 InStock，首页能看到市场概况，进入筛选页可以加载上次保存的条件组合一键扫描，命中的股票可以一键加入关注列表。

### 核心任务

1. **首页市场概览** — 4 卡片布局：市场温度（涨跌比例）、今日扫描命中数、关注列表异动、最近一次扫描时间
2. **筛选条件保存/加载** — 保存当前条件组合为"我的筛选方案"，下次一键加载（消费已有的 /selection/my-conditions CRUD）
3. **关注列表联动** — 结果列表页和详情页增加"加入关注"按钮（消费已有的 attention_router）
4. **扫描历史记录** — 展示最近 N 次扫描的条件和结果摘要，支持重新执行

### 明确不做

- ❌ 回测功能
- ❌ 策略编辑器
- ❌ 性能优化（除非 0a 的基线不达标）

### Phase 0 整体 DoD（进入 Phase 1 的前提）

- [ ] 扫描 → 筛选 → 验证 主线完整可用
- [ ] 首页市场概览可展示
- [ ] 筛选条件可保存/加载/复用
- [ ] 关注列表可从扫描结果中添加
- [ ] 全市场扫描响应时间 < 30s
- [ ] 数据更新连续 10 个交易日稳定
- [ ] 核心 schema 和 service 模块测试覆盖率 > 60%

## Phase 1：策略验证闭环（方向性规划）

### 目标

从"发现标的"进化到"验证假设"——用户不仅能找到符合条件的股票，还能回答"如果过去 1 年按这个条件买入，收益如何？"

### 预期能力

- **参数化策略模板** — 基于已有的 strategy_service 扩展，将筛选条件组合 + 买卖规则封装为可回测的策略
- **回测报告** — 基于已有的 backtest_router 骨架完善，输出收益曲线、最大回撤、胜率等核心指标
- **策略对比** — 多个策略的回测结果并排对比

### 前置约束（Phase 0 必须满足）

- 筛选条件的 schema 格式必须可被回测引擎直接消费（Phase 0a 任务 2 中预留）
- 历史数据深度 ≥ 1 年（Phase 0 数据更新链路需确认）

### 不在 Phase 1 范围

- 实时信号推送
- 自定义策略代码编辑器
- AI 辅助分析

## Phase 2：体验打磨与研究沉淀（方向性规划）

### 目标

在核心链路跑通的基础上，把产品从"能用"打磨到"顺手"。

### 预期能力

- 筛选面板交互优化
- 条件模板与快捷操作
- 多策略 / 多结果对比视图
- 自选 / 观察 / 提醒能力增强
- 最近研究上下文

### 约束

不应滑向：通用笔记系统、项目式研究管理器、复杂任务/看板系统

## 存储与架构演进原则

### 当前策略

- 保留 PostgreSQL + Redis 主栈
- 不在当前阶段做存储层迁移

### 低成本预留

- **数据访问抽象**：扫描/回测引擎通过 MarketDataProvider 接口获取数据
- **每日 Parquet 导出**：盘后任务链路末端增加导出，为 DuckDB / Polars 做准备

### 中期策略

若 Phase 1 后期出现大规模扫描与回测性能瓶颈，在分析层增量引入 DuckDB / Parquet，不替换 PG 主栈。

## 每次迭代检查

### 迭代前

- 这次做的东西服务的是核心链路哪一段？
- 它是否会增强可解释性或可信度？
- 是否存在更小、更直接的做法？

### 迭代后

- 用户完成同一个任务是否更快了？
- 结果是否更清楚、更值得信任了？
- 如果把这个功能删掉，产品会不会反而更清晰？

---

## 最终判断标准

> **先把"扫描 → 筛选 → 验证"做到自己每天都想用，其他一切都是噪音。**
