# InStock 路线图（Roadmap）

## 文档目标

这份路线图用于回答三个问题：

- 接下来应该先做什么，后做什么？
- 每一阶段为什么值得做？
- 如何确保每次迭代没有偏离长期目标？

本路线图默认遵循《PRD.md》中的产品定义：

**InStock 是一个面向个人投资者与研究者的 A 股研究分析工作台。**

它不是资讯站，也不是交易终端，而是帮助用户完成：

- 市场认知
- 条件筛选
- 策略验证
- 研究沉淀
- AI 协作

---

## 0. 路线图的总原则

### 0.1 优先级判断原则

任何需求都按以下顺序判断价值：

1. 是否增强了“研究工作流”的完整性？
2. 是否降低了从“发现问题”到“验证假设”的摩擦？
3. 是否增强了平台的可解释性与可信度？
4. 是否沉淀成可复用能力，而不是一次性页面？
5. 是否为后续 AI 协作铺路？

### 0.2 远期目标校验问题

每完成一个阶段，都必须回头回答：

- 这个阶段是否让平台更像“研究工作台”，而不是功能集合？
- 这个阶段是否让用户更容易理解结果，而不是只多了参数？
- 这个阶段是否强化了专业深度，而不是为了好看牺牲能力？
- 这个阶段是否让未来 AI 更容易接入，而不是更难？
- 这个阶段是否值得用户每天回来打开？

### 0.3 定义完成（Definition of Done）

任何重要功能上线前，至少满足：

- 有明确服务对象与使用场景
- 有可解释的成功标准
- 有可复用的数据/能力结构
- 有失败与异常时的可诊断路径
- 与 PRD 的长期定位一致

---

## 1. 当前项目基线

基于现有仓库，InStock 已经具备较强的后端分析基础：

- FastAPI 后端骨架
- 股票、ETF、指标、策略、回测、选股、资金流、市场等 API 路由
- `core/` 层中已有抓取、指标、K 线、形态、策略等分析模块
- Vue 前端工程可作为现有工作台基础
- 已有测试、CI、Docker、配置和日志能力

这意味着路线图不应该从 0 开始做“另一个新项目”，而应基于已有能力完成两件事：

1. 产品重构：把分散能力组织成清晰工作流
2. 架构重构：把零散服务组织成统一能力层，为前端和 AI 复用

---

## 2. 路线图总览

建议将后续演进拆为 5 个阶段：

- Phase 0：产品与能力基线重构
- Phase 1：工作台与市场/个股分析体验
- Phase 2：筛选与扫描引擎产品化
- Phase 3：策略构建、回测与报告闭环
- Phase 4：个人研究工作台完善（观察、提醒、记录）
- Phase 5：AI 研究助手接入

建议节奏：先做 Phase 0-2，形成可日常使用的研究平台；之后再做 Phase 3-5。

---

## 3. Phase 0：产品与能力基线重构

### 3.1 目标

在不大改技术栈的前提下，完成产品方向与能力边界的重新梳理。

### 3.2 为什么先做这个

如果不先重构基线，后面很容易继续沿着“功能堆叠”的路径走：

- 前端页面越做越散
- API 越加越多，但边界越来越乱
- 后面接 AI 时会发现没有统一工具接口

### 3.3 重点交付

#### 产品侧
- 完成新版 PRD
- 完成路线图和模块边界定义
- 确定一级信息架构
- 确定工作台、股票详情、筛选、回测、研究记录的主流程

#### 技术侧
- 盘点现有 API 与服务能力
- 建立 capability layer 的目录设计与命名规则
- 统一关键结果 schema 设计
- 识别同步请求、定时任务、长任务的边界

### 3.4 具体任务建议

- 梳理现有 router 与 service 的映射关系
- 标记哪些接口是“给页面用”，哪些是“核心能力”
- 定义标准对象：市场摘要、股票摘要、筛选结果、回测结果
- 梳理前端当前页面结构与未来 IA 的差距

### 3.5 阶段退出标准

- 产品方向与模块边界稳定
- 有明确的页面优先级和能力优先级
- 能描述“下一阶段先做哪几个页面，依赖哪些后端能力”

### 3.6 第一性原理回看

- 我们是在让平台更像工作台，还是只是把旧功能换个文档？
- 我们是否把“结果对象”定义清楚了，而不是继续让页面直接消费杂乱接口？

## 4. Phase 1：工作台与市场 / 个股分析体验

### 4.1 目标

让 InStock 成为一个可以每天打开、且不会让人迷路的研究入口。

### 4.2 为什么现在做

一个专业平台如果没有强入口，用户就会只记得“有很多能力”，但不会形成日常使用习惯。

### 4.3 重点交付

#### 页面与体验
- 工作台首页
- 全局搜索
- 个股详情页 v1
- 我的关注 / 最近研究

#### 数据与接口
- 市场概览摘要接口
- 股票搜索接口统一化
- 个股详情聚合接口
- 最近访问 / 最近研究对象存储

### 4.4 页面建议

#### 工作台
- 主要指数概览
- 市场宽度 / 成交额 / 涨跌停概览
- 板块强弱
- 我的关注
- 今日异动
- 最近研究
- 快捷入口

#### 个股详情页 v1
- 基础信息
- K 线 + 成交量
- 核心指标摘要
- 形态识别结果
- 数据更新时间
- 加入观察 / 打开筛选关联入口

#### 全局搜索
- 股票代码 / 名称
- ETF
- 板块主题
- 保存的筛选器和策略
- 最近研究对象

### 4.5 技术任务建议

- 新增 capability：
  - `market/get_overview`
  - `stock/search_assets`
  - `stock/get_detail_summary`
  - `workspace/list_recent_context`
- 建立聚合接口，避免前端页面层拼过多 API
- 为个股详情页建立图表与指标摘要的统一返回结构

### 4.6 阶段退出标准

- 用户可从首页进入完整研究流程起点
- 股票详情页可以支撑基础研究
- 搜索成为统一入口，而不是多个分散输入框
- 平台具备“每天打开看看”的合理性

### 4.7 第一性原理回看

- 这个阶段是否让用户更快理解市场和自己的关注对象？
- 我们是否在减少页面跳转与认知负担？
- 首页是否真的是工作台，而不是数据堆砌？

---

## 5. Phase 2：筛选与扫描引擎产品化

### 5.1 目标

把现有指标、形态、策略相关能力组织成一个真正可用、可保存、可解释的筛选系统。

### 5.2 为什么这是核心阶段

这是 InStock 区别于普通股票信息产品的第一核心壁垒。

用户不会因为“又一个行情页”留下来，但会因为：

- 条件表达足够灵活
- 筛选速度与体验好
- 结果可解释
- 图表能看懂命中原因

而持续使用。

### 5.3 重点交付

#### 产品能力
- 多条件筛选器
- 参数化条件配置
- 范围限制（全市场 / 板块 / 自选组）
- 结果页与结果解释
- 保存筛选器
- 常用模板库

#### 可视化能力
- 图表标注条件命中点
- 筛选结果快速跳转图表
- 结果二次过滤

### 5.4 技术任务建议

- 抽象筛选条件 DSL 或内部表达结构
- 建立 `screener` capability 模块
- 建立筛选结果 schema
- 建立保存筛选器的数据模型
- 支持结果快照和参数快照

### 5.5 阶段退出标准

- 用户能独立创建、保存、复用筛选器
- 用户能解释“为什么命中”
- 筛选结果可直接成为后续分析与策略输入
- 平台开始具备明显的专业分析生产力

### 5.6 第一性原理回看

- 我们是在帮助用户表达研究条件，还是在制造更多复杂表单？
- 我们是否让结果更可理解了？
- 这个阶段的结果对象，未来 AI 能不能直接消费？

---

## 6. Phase 3：策略构建、回测与报告闭环

### 6.1 目标

把“筛选”升级为“可验证的研究方法”，形成策略与回测闭环。

### 6.2 为什么放在这里

如果没有稳定的筛选与结果结构，直接做策略回测 UI 往往会变得混乱、难用、难解释。

### 6.3 重点交付

#### 产品能力
- 低代码策略构建器
- 策略模板
- 回测任务创建与状态管理
- 回测结果页
- 回测报告
- 参数版本对比

#### 分析输出
- 收益率、年化、最大回撤、胜率、盈亏比
- 权益曲线
- 交易明细
- 风险提示
- 基准对比
- 结果摘要解释

### 6.4 技术任务建议

- 引入更明确的任务系统与任务状态持久化
- 建立 `strategy` / `backtest` capability
- 建立回测参数、结果、报告的数据模型
- 将回测任务从同步请求中彻底分离

### 6.5 阶段退出标准

- 用户能构建并保存策略草稿
- 用户能运行回测并查看结构化结果
- 回测结果可复看、可对比、可解释
- 筛选结果与回测之间形成自然衔接

### 6.6 第一性原理回看

- 这个阶段是否真的帮助用户验证假设，而不是只是新增了一个复杂页面？
- 回测结果是否足够支持复盘？
- 我们是在强化“研究平台”身份，还是偏向了花哨但不可复用的结果展示？

---

## 7. Phase 4：个人研究工作台完善

### 7.1 目标

让 InStock 从“分析工具”进化为“可持续使用的个人研究环境”。

### 7.2 重点交付

- 自选股分组
- 观察列表
- 条件命中提醒
- 回测完成提醒
- 研究笔记
- 研究状态流转
- 最近研究 / 历史记录中心

### 7.3 为什么重要

如果没有这一层，用户每次进入平台都像重启。
有了这一层，平台才会开始“记住用户”，形成真正的个人工作台属性。

### 7.4 技术任务建议

- 增加 workspace / watchlist / notes / alerts 相关模型
- 打通筛选器、策略、回测、股票详情与研究记录之间的关联
- 建立统一的任务与提醒中心

### 7.5 阶段退出标准

- 用户可沉淀自己的研究资产
- 用户下次回来可以继续上次工作，而不是从头开始
- 平台支持完整的“发现 → 分析 → 观察 → 提醒 → 复盘”闭环

### 7.6 第一性原理回看

- 我们是否开始真正拥有“个人工作台”的持续性？
- 用户的研究对象、研究记录和研究方法是否被系统记住并复用？

---

## 8. Phase 5：AI 研究助手接入

### 8.1 目标

将 AI 作为平台内的检索层、解释层和编排层接入，而不是独立聊天玩具。

### 8.2 AI 的正确定位

AI 不负责替代分析引擎，不负责提供无依据的投资建议。
AI 负责：

- 检索信息
- 解释结果
- 组织上下文
- 推荐下一步动作
- 将自然语言转为平台内操作意图

### 8.3 第一阶段 AI 能力

#### 检索型
- 查市场概况
- 查股票摘要
- 查最近任务
- 查保存的筛选器与策略

#### 解释型
- 为什么这只股票被筛出来？
- 这次回测的主要问题是什么？
- 这个形态在图上怎么看？

#### 编排型
- 帮我把当前筛选保存为模板
- 在当前结果上再加一个条件
- 创建一个回测草稿

### 8.4 技术任务建议

- 明确 AI tool contract
- 将 capability layer 暴露为 AI 可调用工具
- 为页面注入上下文对象（当前股票、当前筛选、当前回测）
- 增加 AI 工具调用日志与结果引用机制

### 8.5 阶段退出标准

- AI 能基于平台事实回答问题，而不是脱离数据瞎说
- AI 能帮助用户减少操作摩擦
- AI 回答能够引用平台内对象与数据时间上下文

### 8.6 第一性原理回看

- AI 是否真的缩短了研究路径？
- AI 是否建立在平台事实之上？
- AI 是让产品更强，还是只增加了一个看起来很酷的入口？

---

## 9. 跨阶段基础工程（持续推进）

以下工作不应只在某一个阶段做，而应贯穿推进：

### 9.1 统一能力层
- 梳理并收口核心业务能力
- 避免页面和 AI 直接依赖散乱 service

### 9.2 统一结果 schema
- 市场摘要
- 股票摘要
- 筛选结果
- 回测结果
- 研究记录

### 9.3 任务系统与可观测性
- 数据更新任务
- 筛选任务
- 回测任务
- 告警与失败可见
- 数据新鲜度可见

### 9.4 前端设计系统与页面模式统一
- 工作台模式
- 列表 + 过滤 + 详情模式
- 图表 + 摘要 + 下一步动作模式

### 9.5 测试与文档
- 关键能力增加测试
- 关键对象与 API 补文档
- 保证产品文档、接口文档与实现不持续背离

## 10. 推荐执行顺序

如果资源有限，建议采用以下顺序：

### 第一优先级
- Phase 0
- Phase 1
- Phase 2

目标：先把 InStock 做成“能日常打开、能高效筛选、能看懂结果”的平台。

### 第二优先级
- Phase 3
- Phase 4

目标：形成策略验证与研究沉淀闭环。

### 第三优先级
- Phase 5

目标：将 AI 作为成熟平台上的协作层接入，而不是提前透支复杂度。

## 11. 每次迭代前后的检查模板

### 11.1 迭代前检查
- 这次要做的功能，服务于哪个核心用户任务？
- 它属于“市场认知、筛选发现、验证回测、研究沉淀、AI 协作”中的哪一类？
- 它是否会打断已有工作流，还是让工作流更顺？
- 它需要哪些基础能力先标准化？

### 11.2 迭代后检查
- 用户完成同一个任务是否更快了？
- 用户是否更容易理解结果了？
- 功能是否可复用，能否为后续 AI 提供稳定输入？
- 是否引入了新的混乱页面或临时接口？
- 如果删掉这个功能，产品会不会反而更清晰？

---

## 12. 最终判断标准

如果路线走对了，未来的 InStock 应该呈现出下面的样子：

- 用户每天先从工作台进入，快速建立市场认知
- 通过搜索和筛选快速发现值得研究的标的
- 在详情页中看到图表、指标、形态、解释与下一步动作
- 将研究条件沉淀为筛选器和策略
- 用回测验证方法，而不是凭感觉
- 用观察、提醒、记录把研究过程沉淀下来
- 最后让 AI 在平台内帮助检索、解释、编排，而不是替代事实基础

如果做出来的东西偏离了这个方向，就应该及时回滚或重构。
