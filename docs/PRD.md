## 〇、设计哲学（第一性原理）

> 本节是产品决策的底层锚点，不直接映射为功能模块，
> 但当面临"要不要做 X"的决策时，回到这里检验。

投资的本质是利用信息差和认知差获取超额收益。
对个人投资者而言，这个过程可以抽象为：

1. **感知** — 市场现在是什么状态？（牛/熊/震荡）
2. **发现** — 哪些标的出现了值得关注的信号？
3. **验证** — 这个信号在历史上靠谱吗？
4. **跟踪** — 买入后如何持续监控？

InStock 当前阶段聚焦于 **2（发现）和 3（验证）**，
因为这是个人投资者最缺工具支持、且量化手段最能发挥价值的环节。

1（感知）通过首页概览轻量覆盖，4（跟踪）通过关注列表最小覆盖。
两者在 Phase 1+ 逐步深化。

**决策检验模板：**
当考虑新增功能 X 时，问自己：
- X 属于感知/发现/验证/跟踪中的哪一步？
- 它是否强化了"发现+验证"这条当前主线？
- 如果不是，它能否推迟到 Phase 1+？


## 一、产品定位

### 1.1 一句话定义

InStock 是一个**面向个人投资者的 A 股量化筛选与策略验证工具**。

它不是资讯门户，不是看盘终端，不是社区平台。
它的核心价值是：**帮助用户从 5000+ 只 A 股中，用可量化、可解释的方式发现值得关注的标的，并验证交易假设。**

### 1.2 目标用户画像

**主要用户：技术派个人投资者（"老李"）**
- 35 岁，有 5 年以上 A 股投资经验
- 每天盘后花 30-60 分钟做复盘和选股
- 理解 MACD、KDJ、布林带等常用技术指标
- 不会编程，但能理解"金叉"、"放量突破"等条件组合的含义
- 痛点：手动翻看自选股效率低，同花顺/东财的选股器条件不够灵活，且不知道"为什么选出这只"

**次要用户：量化爱好者（"小张"）**
- 28 岁，有 Python 基础，想做量化但没有完整框架
- 希望能快速验证"某个策略在历史上表现如何"
- 痛点：自己搭数据管道太累，只想专注于策略逻辑

> 当前阶段（Phase 0-1）以"老李"为核心用户，UI 设计以"选条件 → 看结果 → 查详情"的直觉操作为主。

### 1.3 核心使用循环

用户每天盘后的使用流程收敛为 **三步主线**：

**扫描（Scan）→ 筛选（Filter）→ 验证（Verify）**

| 步骤 | 用户动作 | 系统能力 |
|------|---------|---------|
| **扫描** | 选择筛选条件组合，点击"开始扫描" | 全市场多条件并行筛选 |
| **筛选** | 浏览命中结果列表，按证据强度排序 | 返回命中列表 + 每条的命中原因摘要 |
| **验证** | 点击个股查看详情，判断是否值得跟踪 | K 线 + 指标叠加 + 形态标注 + 命中证据高亮 |

**闭环动作（贯穿三步）：**
- 保存筛选条件组合（下次一键复用）
- 标记关注个股（加入关注列表）
- 查看历史扫描记录（对比变化）

### 1.4 长期愿景

InStock 的长期愿景是：成为一个面向个人投资者的 A 股研究分析平台。

但当前执行核心必须更收敛：

> **InStock 在当前和未来较长一段时间内，必须优先被当作"面向个人投资者的 A 股量化筛选与策略验证工具"来做。**

### 1.5 产品判断准则

未来任何需求、页面、架构修改，都必须先回答：

1. 它是否增强了"扫描 → 筛选 → 验证"这条核心链路？
2. 它是否缩短了用户从"信息"到"洞察"的路径？
3. 它是否增强了可解释性和可信度？
4. 它是否能沉淀成可复用能力，而不是一次性页面？
5. 它是否会把产品拉向资讯站、交易终端或 AI 荐股器？

如果答案偏负面，应降级或删除。

### 1.6 反方向警报（明确不做）

| 方向 | 为什么不做 |
|------|-----------|
| 资讯门户化 | 新闻/公告/研报不是核心价值，且数据源成本高 |
| 看盘终端化 | 实时行情需要付费数据源，且与同花顺/东财竞争无意义 |
| 社区化 | 个人工具不需要社交功能 |
| 过早 AI 化 | AI 助手/自然语言选股是 Future 方向，当前阶段数据基础和策略体系未稳固 |
| 多用户 SaaS 化 | 当前是个人工具，用户体系仅用于保存个人配置 |
| 实时 tick 级行情推送 | 当前阶段不做 |
| 交易执行 / 接券商 API | 永远不做 |
| 新闻 / 公告 / 研报聚合 | 永远不做 |
| 社区 / 评论 / 排行榜 | 永远不做 |
| AI 荐股 / 黑盒推荐 | 永远不做 |
| 多市场（港股/美股）扩张 | 当前阶段不做 |

## 二、功能架构

### 2.1 功能分层

```
┌─────────────────────────────────────────────┐
│ 展示层（Frontend）                           │
│ 筛选面板 │ 结果列表 │ 个股详情 │ 首页概览    │
├─────────────────────────────────────────────┤
│ 服务层（API）                                │
│ SelectionService │ StockService │ PatternService │
│ IndicatorService │ StrategyService │ BacktestService │
├─────────────────────────────────────────────┤
│ 引擎层（Core）                               │
│ 扫描引擎 │ 形态识别 │ 指标计算 │ 回测引擎    │
├─────────────────────────────────────────────┤
│ 数据层（Data）                               │
│ MarketDataProvider │ DB │ Scheduler          │
└─────────────────────────────────────────────┘
```

### 2.2 "命中证据"定义（核心产品契约）

每条扫描命中结果必须包含**可解释的命中证据**，这是 InStock 区别于传统选股器的核心差异。

**命中证据数据结构示例：**

```json
{
  "stock_code": "600519",
  "stock_name": "贵州茅台",
  "scan_date": "2026-03-31",
  "match_score": 0.85,
  "hit_conditions": [
    {
      "condition_id": "macd_golden_cross",
      "condition_name": "MACD 金叉",
      "description": "DIF 上穿 DEA",
      "evidence": {
        "trigger_date": "2026-03-31",
        "dif_value": 12.35,
        "dea_value": 11.98,
        "prev_dif": 11.80,
        "prev_dea": 12.05
      }
    },
    {
      "condition_id": "volume_surge",
      "condition_name": "成交量放大",
      "description": "当日成交量 / 5日均量 > 1.5",
      "evidence": {
        "trigger_date": "2026-03-31",
        "current_volume": 28500,
        "avg_volume_5d": 18200,
        "ratio": 1.57
      }
    }
  ],
  "summary": "MACD 于 3/31 形成金叉（DIF 12.35 上穿 DEA 11.98），同时成交量放大 1.57 倍"
}
```

**关键约束：**
- 每个 `hit_condition` 必须包含 `evidence` 字段，提供触发该条件的具体数值
- `summary` 字段为人类可读的一句话摘要，前端直接展示
- `match_score` 为 0-1 的综合匹配度，用于排序

## 三、产品范围与边界

### 3.1 必须做（核心范围）

#### A. 数据与更新能力
- A 股核心数据稳定更新
- 日线/必要频率数据完整性与异常检查
- 数据更新时间可见

#### B. 扫描与筛选能力
- 指标筛选
- 形态筛选
- 参数化条件
- 条件组合（AND / OR / NOT）
- 范围限制（全市场 / 指定板块 / 自选组等）
- 结构化结果输出

#### C. 结果解释能力
- 每个命中结果必须解释"为什么命中"
- 支持在图表中展示命中证据
- 支持快速进入个股详情页

#### D. 个股研究能力
- K 线图与成交量
- 常见技术指标叠加
- 形态识别结果
- 数据时间上下文

#### E. 策略验证能力
- 参数化策略模板
- 历史回测
- 报告页（收益、回撤、胜率、基准对比等）
- 条件 / 策略保存与复用

### 3.2 可以做，但必须克制（辅助范围）

#### 轻量工作台 / 首页
可以做，但只能作为核心引擎的投影面，不能扩张为行情门户。

#### 自选 / 跟踪 / 提醒
可以做，但只服务于核心研究链路。

#### AI 助手
必须晚于核心引擎成熟之后接入，且只做研究辅助。

## 四、首页 / 工作台的最小可行设计

### 4.1 首页原则

首页不是一个独立产品模块，而是：

> **核心能力的投影面与导流入口。**

首页上的每一个元素，都必须在 1 次点击内把用户带回：
- 筛选结果
- 个股详情
- 回测 / 策略验证

### 4.2 最小首页模块（只保留 4 个卡片）

#### 卡片 1：市场温度计
展示：上涨/下跌家数、涨停/跌停家数、主要指数涨跌、一句市场情绪摘要
约束：只用盘后或稳定频率数据，不扩张为市场全景页

#### 卡片 2：我的关注
展示：自选股或观察列表 + 其中哪些股票触发了已保存的扫描条件

#### 卡片 3：今日扫描发现（最关键的卡片）
展示：你保存的筛选条件今天命中了多少只股票，一键跳到对应结果页

#### 卡片 4：策略信号 / 回测更新
展示：某个策略新增信号数量、最近一次回测更新摘要（Phase 1 后接入）

### 4.3 首页坚决不做

- 板块热力图
- 龙虎榜 / 大单监控
- 新闻 / 公告 / 研报列表
- 多 tab 仪表盘
- 自定义拖拽布局
- 复杂资金流大盘

## 五、功能模块定义

### 5.1 扫描与筛选引擎（第一核心）

#### 必须支持
- 常见技术指标
- 常见形态识别
- 参数化阈值和周期
- 条件逻辑组合
- 范围过滤
- 结果分页、排序、导出（可后置）

#### 必须输出
- 命中股票列表
- 每只股票的命中原因摘要（参见 2.2 命中证据定义）
- 图表证据可视化入口
- 可保存的筛选条件快照

### 5.2 个股详情（第二核心）

#### 必须支持
- 基础行情摘要
- K 线 + 成交量
- 核心技术指标叠加
- 形态标注
- 当前数据更新时间
- 来自哪些筛选条件的命中说明

### 5.3 策略与回测（第三核心）

#### 当前阶段只做
- 参数化策略模板
- 历史回测
- 回测报告
- 策略参数保存与对比

#### 当前阶段不做
- 通用低代码策略编排器
- 可视化流程编辑器
- 自动交易联动

### 5.4 跟踪与沉淀（辅助核心）

#### 当前阶段只做
- 自选 / 观察分组
- 已保存筛选器
- 已保存策略
- 最近研究上下文
- 必要提醒（可 Phase 2 再强化）

#### 当前阶段不做
- 通用笔记系统
- 复杂项目式研究管理
- 社区化分享沉淀

### 5.5 AI 助手（未来能力）

#### 正确定位
- 检索器、解释器、编排器、摘要器

#### 错误定位
- 选股器、拍板器、替代规则引擎的黑盒分析器

## 六、架构要求与技术约束

### 6.1 总体原则

核心引擎必须独立于 Web 框架存在。真正的资产应是：
- data pipeline、indicator engine、pattern engine、screener engine、strategy / backtest / report engine
- 而不是 FastAPI router 本身

### 6.2 存储与执行策略

- 保留 **PostgreSQL + Redis** 主栈继续迭代
- 不在当前阶段做大规模存储迁移
- A 股当前量级下，PG 足够支撑 Phase 0-1

### 6.3 必须预留的两项能力

#### 预留 1：数据访问层抽象
扫描引擎与回测引擎不要直接耦合 ORM/SQL 查询。应通过 `MarketDataProvider` 等抽象接口获取数据。

#### 预留 2：每日 Parquet 导出
在盘后数据更新链路末端增加 Parquet 导出，为未来引入 DuckDB / Polars 做准备。

### 6.4 前端约束

前端最重要的不是首页，而是：
- 筛选面板交互
- 结果列表浏览效率
- 个股详情的证据可视化
- 策略回测报告的可读性

前端状态应尽可能做到：URL 即状态、刷新不丢上下文、可保存/可复现。

## 七、AI 的边界与接入前提

### 7.1 为什么不能太早做

- 数据与结果未结构化，AI 只会输出流畅废话（金融语境下是误导）
- 尚未积累足够用户使用模式
- 会导致产品定位漂移：从"我设条件，工具帮我找"滑向"AI 告诉我买什么"

### 7.2 AI 接入前提

至少满足：
- 扫描引擎输入输出 schema 化
- 回测报告结构化
- 数据时间上下文可见
- Phase 0-1 完成且稳定运行一段时间

### 7.3 AI 的正确切入点

- 自然语言转筛选条件
- 回测结果解读
- 个股技术面摘要生成
- 下一步操作建议

### Future：AI 研究助手
- Phase 0-1 稳定运行后接入

## 九、成功标准

### Phase 0
用户（首先是开发者本人）能够：
- 每天盘后打开 InStock
- 快速运行筛选条件
- 看到命中结果，理解命中证据
- 觉得它比直接去资讯终端多出真实价值

### Phase 1
用户能够：
- 把筛选方法转成参数化策略
- 跑历史回测
- 明确理解收益、风险和基准对比

### 长期
InStock 真正变成每日研究入口、个人研究方法的承载工具、可解释可信任的筛选与验证平台。

## 十、最终判断标准

> **先把"扫描 → 筛选 → 验证"这条管道做到自己每天都想用，其他一切都是噪音。**
