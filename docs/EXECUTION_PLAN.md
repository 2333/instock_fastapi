# InStock 执行计划（Execution Plan）

## 文档目标

把《PRD.md》与《ROADMAP.md》中的 Phase 0 / Phase 1 落成可执行任务。

原则：

- 第一性原理优先
- 先做能形成真实使用价值的链路
- 不为理想架构提前支付过大成本
- 每个任务都尽量服务“扫描 -> 筛选 -> 验证”主线

---

## 0. 当前执行优先级

按绝对优先级排序：

1. 数据更新稳定性
2. 扫描引擎 API 化
3. 命中证据可视化
4. 个股详情页
5. 参数化策略模板 + 回测报告

任何会明显抢占以上事项资源的工作，应降级。

---

## 1. Phase 0 执行计划：可用基线

### 1.1 目标

形成一个每天盘后可真实使用的最小可用产品：

- 能输入筛选条件
- 能看到结果
- 能打开个股详情
- 能看懂命中证据

### 1.2 工作流级定义完成（DoD）

当以下流程能顺畅跑通时，Phase 0 才算完成：

1. 数据更新任务运行成功
2. 用户配置 / 选择筛选条件
3. 扫描引擎返回结构化结果
4. 结果页可查看命中原因
5. 点进个股详情页可看到 K 线、指标、形态与命中证据
6. 首页能以 4 卡片形式摘要这些结果

---

### 1.3 后端任务

#### P0-Backend-01：梳理数据更新链路
- 盘点当前数据源、任务、失败点
- 明确日线更新、指标计算、形态计算的顺序
- 为关键任务补充统一日志和失败信号

#### P0-Backend-02：定义扫描引擎输入输出 schema
- 定义筛选条件模型
- 定义结果项模型
- 定义命中原因 / 命中证据结构
- 使用 Pydantic 固化边界

#### P0-Backend-03：将筛选逻辑 API 化
- 把现有指标 / 形态筛选逻辑收敛到统一入口
- 支持参数化输入
- 支持范围限制
- 支持组合逻辑

#### P0-Backend-04：建立结果证据结构
- 为每条结果输出命中条件摘要
- 为个股详情页提供图表标注数据
- 明确“为什么命中”的解释对象

#### P0-Backend-05：建立个股详情聚合接口
- 基础行情摘要
- K 线与成交量数据
- 常见技术指标数据
- 形态结果
- 数据更新时间
- 命中来源说明

#### P0-Backend-06：最小首页数据接口
- 市场温度计摘要
- 我的关注摘要
- 今日扫描发现摘要
- 策略信号占位接口（可先返回空）

---

### 1.4 架构任务

#### P0-Arch-01：引擎与 API 解耦
- 明确 engine / service / router 责任边界
- 避免 router 直接承担业务拼装逻辑

#### P0-Arch-02：引入数据访问抽象
- 为扫描 / 详情 / 回测预留 MarketDataProvider 接口
- 当前实现仍走 PostgreSQL
- 不做大迁移，只做解耦起点

#### P0-Arch-03：每日 Parquet 导出预留
- 在盘后任务链路末尾增加导出能力
- 先做到可运行、可验证，不要求立刻消费

---

### 1.5 前端任务

#### P0-Frontend-01：筛选页 v1
- 条件输入区
- 参数配置区
- 运行按钮
- 结果页入口

#### P0-Frontend-02：结果列表页 v1
- 列表展示
- 排序 / 基础过滤
- 每行显示命中原因摘要
- 一键进入个股详情页

#### P0-Frontend-03：个股详情页 v1
- K 线 + 成交量
- 常见指标叠加
- 形态结果
- 命中证据展示
- 数据更新时间

#### P0-Frontend-04：首页 4 卡片
- 市场温度计
- 我的关注
- 今日扫描发现
- 策略信号占位

> 首页放在 Phase 0 最后，不得优先于筛选结果页和详情页。

---

### 1.6 数据与质量任务

#### P0-Data-01：数据更新时间可见
- 前端显示更新时间
- 后端明确数据来源时间和计算时间

#### P0-Data-02：异常与缺失检查
- 关键任务失败可发现
- 数据更新缺失可识别
- 结果页尽量避免基于未知状态的数据输出

#### P0-Data-03：最小质量基线
- 至少抽样检查若干常见指标结果是否合理
- 至少检查若干形态输出是否与图表大体一致

---

### 1.7 测试任务

#### P0-Test-01：扫描引擎输入输出测试
#### P0-Test-02：关键结果 schema 测试
#### P0-Test-03：数据更新关键链路 smoke test
#### P0-Test-04：个股详情聚合接口测试

---

### 1.8 建议执行顺序

1. 数据更新链路梳理
2. 扫描 schema 与 API 化
3. 个股详情聚合接口
4. 结果证据结构
5. 前端筛选页 / 结果页 / 详情页
6. 首页 4 卡片
7. 测试与 dogfooding

## 2. Phase 1 执行计划：策略验证闭环

### 2.1 目标

用户能把筛选方法转成参数化策略，运行历史回测，并理解结果。

### 2.2 工作流级定义完成（DoD）

1. 用户选择或保存一个策略模板
2. 配置参数并发起回测
3. 回测作为任务运行并返回结果
4. 用户查看收益、风险、基准对比与解释摘要
5. 用户可以保存、复看、比较不同参数版本

---

### 2.3 后端任务

#### P1-Backend-01：参数化策略模板模型
- 定义策略模板
- 定义参数结构
- 定义模板输入校验

#### P1-Backend-02：回测任务模型
- 创建回测任务
- 跟踪状态
- 保存参数快照
- 保存结果快照

#### P1-Backend-03：回测引擎接入
- 优先基于现有成熟框架封装
- 避免从零实现底层回测引擎

#### P1-Backend-04：回测报告结构化输出
- 收益曲线
- 最大回撤
- 胜率
- 盈亏比
- 基准对比
- 关键风险摘要

#### P1-Backend-05：策略保存与 URL 化
- 条件 / 参数保存
- 可复现链接
- 版本对比基础能力

---

### 2.4 前端任务

#### P1-Frontend-01：策略模板页 / 配置页
- 选择模板
- 配置参数
- 运行回测

#### P1-Frontend-02：回测结果页
- 收益曲线
- 关键统计指标
- 与基准比较
- 结果摘要说明

#### P1-Frontend-03：策略/参数对比页（最小版）
- 同一策略不同参数版本对比
- 核心指标对比表

---

### 2.5 架构任务

#### P1-Arch-01：任务系统规范化
- 回测任务与普通请求分离
- 状态持久化
- 错误可见

#### P1-Arch-02：结果对象继续 schema 化
- 回测报告 JSON 输出固定结构
- 为未来 AI 解读预留契约

---

### 2.6 测试任务

#### P1-Test-01：策略模板参数校验测试
#### P1-Test-02：回测任务状态流测试
#### P1-Test-03：回测报告字段与结果结构测试

---

## 3. 暂不进入执行的事项

以下事项现在不要进入主任务板：

- 低代码策略拖拽编排器
- 社区 / 分享 / 排行榜
- 资讯聚合
- 复杂笔记系统
- 移动端优先改造
- AI 聊天入口开发
- DuckDB 全面迁移

这些都应等核心链路形成真实稳定价值后再判断。

---

## 4. 每次迭代自检问题

在每一轮开发开始前和结束后，都问这几个问题：

1. 这次是否让“扫描 -> 筛选 -> 验证”更顺了？
2. 这次是否让结果更可信、更可解释了？
3. 这次是否只是让产品看起来更像个平台，而不是更好用？
4. 这次是否为未来 AI 或分析层优化留下了低成本接口？
5. 如果把这项工作删掉，产品会不会反而更聚焦？

---

## 5. 最后一条执行原则

> **优先做自己会每天打开用的功能，而不是会让路线图看起来更完整的功能。**
